## 4. Flow Audit Result

`FatigueSlotPolicy` was already used in long-term program generation:

```text
ProgramBuilder.build(...)
-> fatigueSlotPolicy.gate(fatigueState: DailyFatigueState?)
-> ProgramTemplateCatalog.weekPlans(...)
-> schedule adaptation
-> candidate filtering
-> prescription volume/RPE shaping
```

That path still uses legacy `DailyFatigueState?`. It was not changed to accept `TrainingGateSnapshot?`.

The current-day injection boundary chosen for this step is:

```text
TrainingViewModel.applyProgram(...)
-> TrainingRepository.applyProgramToDates(...)
-> dateForProgramItem(startDate, item)
-> if resolved item date == today, apply TrainingGateSnapshot through FatigueSlotPolicy
-> insert adjusted WorkoutEntry / WorkoutSet rows
```

Reason:

```text
applyProgramToDates is where stored TrainingProgramItem rows become dated planned workout rows.
At that point the code knows the actual calendar date for each item.
This allows today's gate to affect only today's inserted session without mutating the stored multi-week program.
```

## 5. TrainingGateSnapshot Source

`TrainingViewModel.applyProgram(...)` now passes only the gate snapshot:

```kotlin
trainingGate = _phaseAwareTodayStatus.value?.current?.fatiguePresentation?.gate
    ?: _todayReadinessSummary.value?.fatiguePresentation?.gate
```

It does not pass the whole `TodayReadinessSummary`.

Fallback order:

```text
1. Phase-aware current summary fatiguePresentation.gate
2. Today readiness summary fatiguePresentation.gate
3. null
```

If the gate is null, repository behavior remains the legacy path.

## 6. ProgramFatigueGate Extension

`FatigueSlotPolicy` now supports:

```kotlin
fun gate(trainingGate: TrainingGateSnapshot?): ProgramFatigueGate
```

This reuses the existing `ProgramFatigueGate` model instead of introducing another engine or taxonomy.

Direct field mapping:

```text
TrainingGateSnapshot.volumeFactor -> ProgramFatigueGate.volumeFactor, clamped to 0.0..1.0
TrainingGateSnapshot.rpeCap -> ProgramFatigueGate.rpeCap, clamped to 1..10, default 9 when null
heavyLowerRestricted -> allowsHeavyLower = false, lowerBodyRestricted = true
highImpactRestricted -> allowsHighImpact = false
codReactiveRestricted -> allowsHighIntensityCod = false
upperPushRestricted -> allowsUpperPush = false
overheadRestricted -> allowsOverhead = false
gripForearmRestricted -> allowsGripForearm = false
```

No `gate.reasons` strings are parsed.

No new fatigue thresholds were added. For the snapshot-derived gate, `ProgramFatigueBand` is only set to:

```text
GREEN when there is no typed restriction and volumeFactor is 1.0
YELLOW when any typed restriction exists or volumeFactor is below 1.0
```

The current-day adjustment logic uses the typed booleans and `volumeFactor`, not duplicated severity thresholds.

## 7. Current-Day Adjustment Algorithm

New policy method:

```kotlin
fun adjustTodayItem(
    item: TrainingProgramItem,
    candidate: ProgramCandidate?,
    gate: ProgramFatigueGate
): TrainingProgramItem?
```

Behavior:

```text
Return the same item when no restriction applies and volumeFactor is 1.0.
Return a copied item with reduced setCount when volume scaling or downgrade applies.
Return null when today's item should be avoided.
Never mutate the stored TrainingProgramItem.
Never increase setCount.
Keep at least 1 set for kept/downgraded items.
```

Volume scaling:

```text
original setCount is coerced to at least 1
effective factor is clamped to 0.0..1.0
scaled setCount is rounded and clamped to 1..original
forced downgrade sets setCount to 1
```

Example:

```text
3 sets with volumeFactor 0.5 -> 2 sets
4 sets with heavyLowerRestricted and matching lower slot -> 1 set
high impact slot with highImpactRestricted -> skipped for today
```

## 9. Repository Boundary Details

`TrainingRepository.applyProgramToDates(...)` now has a nullable parameter:

```kotlin
trainingGate: TrainingGateSnapshot? = null
```

When the parameter is null:

```text
No readiness gate context is built.
No exercise metadata is loaded for gate adjustment.
Existing insertion behavior is preserved.
```

When the parameter is non-null:

```text
today = SystemAnalysisDateProvider().today()
exercises = exerciseDao.allExercises()
metadataCatalog = resolvedRuntimeMetadataCatalog(exercises)
candidatesByExerciseId = ProgramCandidate per exercise using existing SlotCapabilityResolver
```

During item insertion:

```text
itemDate = dateForProgramItem(startDate, item)
if itemDate == today:
    adjust via FatigueSlotPolicy.adjustTodayItem(...)
else:
    insert the item unchanged
```

If `adjustTodayItem(...)` returns null, only that current-day item is skipped. The stored `TrainingProgramItem` remains unchanged.

## 10. Multi-Week Program Generation Is Not Globally Altered

The following was intentionally not done:

```text
No TrainingGateSnapshot parameter was added to ProgramBuilder.build(...)
No TrainingGateSnapshot parameter was added to ProgramSkeletonGenerator.generate(...)
No ProgramTemplateCatalog gate thresholds were changed
No long-term generated week plan is rewritten based on today's gate
```

Existing `ProgramBuilder` behavior still depends on:

```kotlin
fatigueState: DailyFatigueState? = null
```

The new focused test verifies that `ProgramBuilder.build(...)` does not expose `TrainingGateSnapshot` as a method parameter.

## 11. RPE Cap Decision

`TrainingGateSnapshot.rpeCap` is mapped into `ProgramFatigueGate.rpeCap`, but it is not applied to inserted planned workout rows in this step.

Reason:

```text
TrainingProgramItem has no targetRpe or plannedRpe field.
WorkoutEntry.rpe and WorkoutSet.rpe are recorded-performance fields, not planned target RPE fields.
```

Therefore, capping those fields would risk overwriting future recorded effort semantics.

The test file includes a guard documenting that `TrainingProgramItem` currently has no planned target-RPE field to cap.

Future safe wiring point:

```text
Add an explicit planned target RPE field to the resolved-session model or program item model.
Then cap only that planned target field, never recorded RPE.
```

## 12. Fallback Behavior

Null gate:

```text
Legacy applyProgramToDates behavior is preserved.
```

Non-null gate but missing candidate metadata:

```text
The item cannot be classified by typed slots.
volumeFactor still applies because it is not taxonomy-dependent.
No exercise-name string matching is used.
```

No replacement pool:

```text
At applyProgramToDates time, the code is resolving stored program items, not regenerating a candidate program.
For restricted slots, the safe fallback is either current-day set downgrade or current-day skip.
No permanent program mutation occurs.
```

## 15. Notes For The Next Step

Do not extend `analysis/fatigue` for this path. The source remains:

```text
analysis/readiness
-> FatiguePresentationSnapshot
-> TrainingGateSnapshot
```

If future work adds a resolved-session model with planned target RPE, then wire:

```text
TrainingGateSnapshot.rpeCap -> planned target RPE cap
```

Do not wire it to:

```text
WorkoutEntry.rpe
WorkoutSet.rpe
```

Those are recorded effort fields.

If future work needs substitutions instead of skip/downgrade, add them at a boundary with access to a candidate exercise pool. `applyProgramToDates(...)` currently has stored program items and metadata classification, not a full replacement search.
