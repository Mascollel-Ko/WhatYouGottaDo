# v0.4.2.9 Release Notes

v0.4.2.9 unifies Home and Record daily-condition editing, adds optional daily
body weight, and makes the LOCAL_MUSCLE OFI detail identify fatigued muscles.
Existing fatigue, readiness, connective-tissue, and program calculations are
unchanged.

## Baseline

- Started from local/main and `origin/main` commit
  `1090eb6a0d4beb144e7a3cefd369138c06b25fe9`, the verified v0.4.2.8 release.
- `v0.4.2.8` and `origin/main` pointed to that same commit.
- Preserved the six pre-existing user-owned `outputs/*` modifications without
  editing, staging, formatting, or committing them.

## Daily Condition

- `DailyCheckIn` is the one canonical date-keyed daily-condition record.
- Home and Record use the same `DailyConditionEditorDialog`, repository stream,
  and full-record save action.
- Home edits the current local date. Record edits its selected local date and
  uses a date-specific title for historical dates.
- The editor contains sleep, optional `bodyWeightKg: Double?`, and all existing
  subjective condition fields in one form.
- Body weight accepts decimal point or comma input, preserves decimal
  precision, allows null/clearing, and rejects zero, negative, NaN, and
  infinite values.
- Form state uses `rememberSaveable`; dismissing does not persist partial
  edits.

## Storage And Compatibility

- Room schema 20 adds nullable `bodyWeightKg` to `daily_check_ins`.
- Migration 19 to 20 preserves all non-null legacy values. Conflicting sleep
  values use the reliably later `updatedAt`; the existing canonical check-in
  wins ties and missing values are filled from `daily_metrics`.
- `DailyMetric` remains a transactionally synchronized compatibility
  projection because existing analysis and body-weight effective-load
  consumers already read it.
- Restore and daily-timeseries imports write through `DailyStatusService`;
  the former Record-only metric write path is no longer exposed to UI or
  ViewModel callers.
- New backup round trips preserve decimal daily body weight. Older backups
  without it remain valid.
- Daily body weight does not add or change any OFI, readiness, volume,
  ProgramBuilder, calorie, connective-tissue, or recovery calculation.

## LOCAL_MUSCLE Presentation

- The card title is now `주의할 피로 축과 주요 내용`.
- LOCAL_MUSCLE uses the secondary label `피로한 근육`.
- It reuses the existing readiness `highBodyParts` authority, ranked by current
  local-fatigue score with stable-key tie-breaking, deduplication, and a
  maximum of three Korean muscle names.
- Its empty text is `두드러지게 피로한 근육이 없습니다.`
- LOCAL_MUSCLE no longer presents exercise names as its primary detail.
- The other four OFI axes retain their existing contributing-exercise details.
- LOCAL_MUSCLE numeric value, status, recovery, five-axis classification,
  thresholds, and exact canonical OFI messages are unchanged.

## Remaining Limitations

- Daily body weight continues through the established `DailyMetric` consumer
  contract only; this release introduces no new weight-based calculation.
- LOCAL_MUSCLE shows the existing meaningful ranked body-part pressures. It
  does not add a muscle-to-exercise drill-down or a new fatigue model.
- Device/emulator execution of the Room migration test remains a CI or
  instrumentation-run responsibility; its Android-test source compiles.
