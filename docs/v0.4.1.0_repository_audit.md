## Risk Assessment

### Low-risk next extraction candidates

- `SmashSpeedService`
  - Move `observeSmashSpeedsForDate`, `addSmashSpeed`, and `deleteSmashSpeed`.
  - Reason: small DAO-only responsibility with no transaction complexity.
  - Tests: smash speed record add/delete and Lab metric regression if available.
- Thin facade read-only query cleanup
  - Keep as-is unless repository size is the only concern. Extraction adds little value.

### Medium-risk candidates

- `CalendarRecordService`
  - Move `calendarConflictSummary`, date delete/range delete, copy/move/range copy, `copyEntriesToDate`, `dateRange`, `nextCreatedAt`.
  - Reason: bounded responsibility still in repository.
  - Risk: confirmed/unconfirmed status, overwrite behavior, completed timestamps, display order.
  - Tests: calendar copy/move/delete focused tests, selected-copy status preservation, existing conflict mode behavior.
- `AnalysisSummaryService`
  - Move `fatigueAnalysisHistory`, `badmintonTransferSummary`, and `badmintonTransferCoverageSummary`.
  - Reason: read-only summary assembly similar to prior home/performance extractions.
  - Risk: effective metadata resolver and date-window semantics.
  - Tests: fatigue history, badminton transfer, coverage analyzer regressions.

### High-risk / do not touch yet

- Restore/import internals.
  - Reason: backup restore is a data-loss boundary and still owns legacy compatibility, duplicate detection, seed/custom exercise reconciliation, runtime metadata override restore, daily/check-in merge rules, and imported exercise fallback creation.
- Exercise metadata editor/override persistence.
  - Reason: this is user-visible and feeds fatigue, muscle load, badminton transfer, Lab metrics, and backup/restore.
- Seed/bootstrap refresh logic.
  - Reason: runs at startup and can overwrite or preserve seed/custom metadata depending on override state.
- ProgramBuilder / ProgramBuilderValidation.
  - Reason: v0.4.1.0 intentionally did not change generated program behavior.
