# v0.4.1.10.6 Program Builder Fatigue and Exercise Selection Release Notes

## Baseline

- Started from: `v0.4.1.10.5`
- Release version: `versionName = "v0.4.1.10.6"`, `versionCode = 401016`

## Cause

ProgramBuilder still mixed future planning fatigue with today execution safety gates. Under high fatigue, this could remove foundation anchors such as squat/hinge/row/press before the builder had a chance to reduce sets, RPE, or load. Exercise avoid/prefer inputs were also memo text instead of actual selected exercises, so builder selection could not safely enforce real exclusions or boost preferred exercises.

## Changes

1. Added focused tests for the fatigue gate and exercise preference gaps.
2. Split fatigue use cases into `PROGRAM_PLANNING` and `TODAY_EXECUTION`.
3. Kept foundation anchors available during planning fatigue and downscaled their dose.
4. Kept ORANGE/RED planning fatigue from selecting explosive transfer work while still allowing downscaled foundation anchors.
5. Reworded planning fatigue messages as adjustment text, not hard-ban text.
6. Added selected excluded/preferred stableKey fields to `ProgramSkeletonRequest`.
7. Applied selected excluded stableKeys as hard candidate excludes and selected preferred stableKeys as reranking boosts.
8. Added search selection UI for avoided and preferred exercises in the auto-skeleton options.
9. Added post-optimization summary messages for selected exercise constraints.

## Behavior

- Future program planning keeps foundation anchors in the candidate pool under fatigue.
- ORANGE/RED planning fatigue still filters explosive transfer work.
- Today execution safety still hard-blocks or downgrades unsafe items when applying a program for today.
- High fatigue planning lowers set count, RPE cap, and weight suggestion intensity for heavy foundation work.
- Free-text excluded exercise memo is preserved, but selected stableKeys are now the reliable hard exclude path.
- Preferred exercises are soft boosts and do not bypass equipment, safety, slot, or session rules.
- Search selection is transient to generation requests; no Room schema change was made.

## Remaining Risks

- Exercise selection UI is intentionally minimal and local to the auto-skeleton editor.
- Preferred exercise boosting is soft; a preferred movement can still be skipped when it violates constraints or does not fit available slots.
- Existing saved programs do not persist selected avoid/prefer stableKeys.

## Follow-up Candidates

- Persist selected avoid/prefer exercise constraints if users expect them to survive edit sessions.
- Prevent the same exercise from being selected as both avoided and preferred.
- Add richer search affordances if the exercise list becomes difficult to scan.
