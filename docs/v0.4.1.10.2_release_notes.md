# v0.4.1.10.2 Release Notes

## Cause

ProgramBuilder was shrinking candidate pools too early. Basic eligibility, slot capability, template matching, fatigue gates, session constraints, scoring, variety, and time budget trimming were handled in one tight flow, and each slot kept only the top three scored candidates before broader variety could work. Runtime metadata slot resolution also truncated candidate permissions, so compound or fallback-capable exercises could lose selection rights.

## User Request

The user reported that automatically generated program skeletons could collapse to only one to four exercises and asked for the candidate permission / selection structure to be cleaned up rather than padding programs with random exercises.

## Changes

- Added `ProgramCandidateInventory` for active/selectable/equipment/excluded candidate collection.
- Added `ProgramSlotCandidateQuery` for slot-stage candidate counts, adaptive selection pools, and required-slot relaxation.
- Added `ProgramCandidateTrace` to generated skeletons.
- Preserved selection rights for all resolved slot capabilities while keeping primary/secondary/weak display buckets.
- Allowed runtime metadata to add legacy/name fallback capabilities when runtime slot matches are missing or too narrow.
- Replaced fixed top-three slot pools with adaptive candidate pools.
- Softened optional time-budget behavior: optional prescriptions are reduced first, then only that slot is skipped if it still does not fit.
- Added seed-based regression tests for candidate collapse and capability fallback.

## Reason

The builder should keep hard gates limited to real eligibility blockers, preserve viable fallback candidates, and apply variety/time decisions only after the candidate pool is wide enough to avoid collapse.

## Result

Auto-generated programs keep broader candidate pools, required slots can relax to suitable fallback candidates, compound exercises keep multiple slot permissions, support/accessory/prehab work is less likely to disappear behind time trimming, and slot candidate shrinkage can be inspected through traces.

## Remaining Risk

- Candidate trace is model-level only; it is not exposed in UI.
- Required slot relaxation is conservative and still respects role/session/fatigue gates.
- Seed metadata quality still determines how rich candidate pools can be.
