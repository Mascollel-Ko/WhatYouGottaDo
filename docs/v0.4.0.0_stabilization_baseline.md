# v0.4.0.0 Stabilization Baseline

v0.4.0.0 is a stabilization baseline release. It consolidates the latest `main`
state after late v0.3.9.x fixes, rather than introducing a broad feature rewrite.

한국어 요약: v0.4.0.0은 대형 신규 기능 릴리스가 아니라, v0.3.9.x 후반의
기록 복사, 메타데이터 override, 근육군 key 정규화, 분석 범례 수정 작업을
통합한 안정화 기준점 릴리스다.

## Baseline Scope

Future work should start from latest `origin/main` after this baseline, not from
an older release tag.

Included late v0.3.9.x fixes:

- Preserve-state selected copy for record ranges.
- Exercise metadata override resolution before analysis.
- Muscle key normalization for load analysis, including quadriceps override cases.
- Muscle trend legend improvement.

## Known Architecture Risks

### TrainingRepository God Object Risk

`TrainingRepository.kt` is about 2,024 lines and contains more than 100 function
or class-like declarations. Record, analysis, backup, metadata, fatigue, and
program responsibilities are concentrated in one repository surface.

Risk:

- Small changes can create wide regression risk.
- Callers can bypass newer resolved-metadata paths by using older repository
  helpers.
- Backup/restore and analysis behavior are harder to verify in isolation.

Future candidate:

- Split repository responsibilities into smaller services or use-cases while
  preserving the current public behavior.

### Large Compose Screens

`RecordScreen.kt` is about 1,974 lines and `PlanScreen.kt` is about 1,378 lines.
They mix UI state, event handling, rendering, and domain calls in large files.
Analysis screen files should remain on the watch list as more chart and Lab UI
is added.

Risk:

- UI fixes can accidentally change domain behavior.
- Bottom inset, dialog, and selector regressions are harder to localize.
- Focused UI tests are harder to add.

Future candidate:

- Extract screen sections, action handlers, and state holders incrementally.

### Analysis/Data Path Coupling

Metadata override, seed metadata, resolved metadata, backup/restore, Lab series,
and muscle-load analysis paths are tightly coupled.

Risk:

- Analysis can regress if any path reads seed metadata directly instead of
  effective resolved metadata.
- UI labels or raw strings can leak into analysis keys.
- Canonical muscle key normalization can be bypassed by new code.

Future candidate:

- Centralize resolved exercise metadata lookup and canonical key normalization
  tests.

### Release Workflow Risk

The `v0.3.9.9` tag is behind current `main`: later main-only fixes include
muscle-key normalization and the muscle trend legend. Starting future work from
an older tag can drop those fixes.

Rule:

- Start new work from latest `origin/main`.
- Do not move pushed tags.
- Create new release tags only on the final verified `main` commit.

## Future Refactor Candidates

1. Split `TrainingRepository` responsibilities into smaller services/use-cases.
2. Extract `RecordScreen` sections and action handlers.
3. Extract `PlanScreen` program-building UI/state logic.
4. Centralize resolved exercise metadata lookup and canonical key normalization tests.
5. Add a release-baseline checklist to prevent tag/main divergence mistakes.
