## 목적

Home 화면의 fatigue summary/card가 readiness pipeline에서 만든 공통 피로 표현값인
`TodayReadinessSummary.fatiguePresentation`을 우선 사용하도록 연결했다.

이 작업은 Home UI 재설계가 아니다.
카드 레이아웃, 차트, ProgramBuilder, `FatigueSlotPolicy`, `analysis/fatigue` 계산 엔진은 건드리지 않았다.

## 변경 전 상태

Home 화면은 다음 경로로 오늘 요약을 표시했다.

1. `HomeScreen`
2. `TrainingViewModel.homeTodaySummary`
3. `TrainingRepository.homeTodaySummary(todayStatus)`
4. `HomeTodaySummaryState`
5. `HomeFatigueCardSummaryFactory.create(...)`
6. `HomeFatigueCardSummary`

변경 전 Home fatigue card의 점수는 `DailyFatigueState`의 legacy fatigue 값에서 왔다.

주요 legacy source:

- `DailyFatigueState.overallFatigueIndex`
- `DailyFatigueState.readinessLabel`
- `DailyFatigueState` 내부 axis score

`TrainingRepository.homeTodaySummary(...)`는 이미 `PhaseAwareTodayStatus?`를 받았고,
`PhaseAwareTodayStatus.current`와 `PhaseAwareTodayStatus.projected`는 각각 `TodayReadinessSummary`를 들고 있었다.

즉 Home까지 readiness summary가 이미 전달되고 있었기 때문에 새 ViewModel state, 새 repository API,
새 fatigue engine은 필요하지 않았다.

## 변경 후 상태

변경 위치:

`app/src/main/java/com/training/trackplanner/analysis/fatigue/HomeFatigueCardSummaryFactory.kt`

`HomeFatigueCardSummaryFactory.create(...)` 안에서 `todayStatus.current.fatiguePresentation` 또는
`todayStatus.projected.fatiguePresentation`이 있으면 그 값을 Home card reading의 우선 source로 사용한다.

현재 reading:

```kotlin
val primaryReading = todayStatus?.current?.let { summary ->
    summary.fatiguePresentation?.toReading(summary.status.homeLabel())
        ?: primaryState.toReading(summary.status.homeLabel())
} ?: primaryState.toReading()
```

Projected reading:

```kotlin
val projectedReading = projected?.let { state ->
    todayStatus?.projected?.let { summary ->
        summary.fatiguePresentation?.toReading(summary.status.homeLabel())
            ?: state.toReading(summary.status.homeLabel())
    } ?: state.toReading()
}
```

새 helper:

```kotlin
private fun FatiguePresentationSnapshot.toReading(label: String): HomeFatigueReading =
    HomeFatigueReading(
        score = overallScore.coerceIn(0, 100),
        label = label
    )
```

기존 helper는 유지:

```kotlin
private fun DailyFatigueState.toReading(labelOverride: String? = null): HomeFatigueReading =
    HomeFatigueReading(
        score = overallFatigueIndex,
        label = labelOverride ?: qualitativeLabel()
    )
```

## Fallback 규칙

필수 fallback 규칙을 그대로 적용했다.

```text
Use fatiguePresentation when non-null.
Otherwise use the existing Home fatigue source exactly as before.
```

세부 동작:

- `todayStatus == null`: 기존 `DailyFatigueState.toReading()` 사용
- `todayStatus.current.fatiguePresentation != null`: readiness `overallScore` 사용
- `todayStatus.current.fatiguePresentation == null`: 기존 current/preWorkout `DailyFatigueState` 점수 사용
- `todayStatus.projected.fatiguePresentation != null`: projected readiness `overallScore` 사용
- `todayStatus.projected.fatiguePresentation == null`: 기존 projected `DailyFatigueState` 점수 사용

## 라벨 매핑

`ReadinessStatus` enum은 이 브랜치에서 다음 값이다.

```kotlin
enum class ReadinessStatus {
    READY,
    CAUTION,
    FATIGUED,
    LIMITED
}
```

사용자가 예시로 준 `REDUCE`, `REST`는 현재 브랜치에 존재하지 않는다.
따라서 같은 의미로 다음처럼 매핑했다.

```kotlin
private fun ReadinessStatus.homeLabel(): String = when (this) {
    ReadinessStatus.READY -> "진행 가능"
    ReadinessStatus.CAUTION -> "주의"
    ReadinessStatus.FATIGUED -> "감량 권장"
    ReadinessStatus.LIMITED -> "휴식 권장"
}
```

라벨 변경 요약:

- `READY`: `진행 가능`
- `CAUTION`: `주의`
- `FATIGUED`: `감량 권장`
- `LIMITED`: `휴식 권장`

이 작업에서는 `FatiguePresentationSnapshot.reasons`, `highCategories`, `highBodyParts`,
`reduceToday`, `availableToday`를 Home UI에 새로 노출하지 않았다.
따라서 raw enum name이 Home에 노출되는 새 경로도 만들지 않았다.

## 테스트 변경

변경 위치:

`app/src/test/java/com/training/trackplanner/analysis/fatigue/HomeFatigueCardSummaryFactoryTest.kt`

추가/수정한 테스트:

1. `readinessFatiguePresentationIsPreferredForPrimaryReading`
   - current readiness summary에 `fatiguePresentation`이 있으면 legacy current score 대신
     `FatiguePresentationSnapshot.overallScore`를 사용하는지 확인.

2. `nullReadinessFatiguePresentationFallsBackToLegacyState`
   - `fatiguePresentation == null`이면 기존 `DailyFatigueState.overallFatigueIndex`를 유지하는지 확인.

3. `readinessFatiguePresentationScoreIsClamped`
   - snapshot score가 100을 넘으면 Home reading score가 `0..100`으로 clamp되는지 확인.

4. `readinessFatiguePresentationIsPreferredForProjectedReading`
   - projected readiness summary에 `fatiguePresentation`이 있으면 legacy projected score 대신
     projected snapshot score를 사용하는지 확인.

기존 테스트 라벨 기대값은 새 `homeLabel()` 매핑에 맞게 수정했다.
