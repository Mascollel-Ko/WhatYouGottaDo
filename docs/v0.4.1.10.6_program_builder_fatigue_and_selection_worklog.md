## Commit 2 - fix(program): separate planning fatigue from today execution gates

수정 원인:
- ProgramBuilder가 multi-week planning에서도 오늘 실행용 fatigue hard gate를 사용하면 foundation anchor 후보가 사라질 수 있다.
- 오늘 실제 적용 경로의 hard safety gate와 미래 계획 생성의 조정 gate는 의미가 다르다.

수정 골자:
- `ProgramFatigueUseCase`를 추가해 `PROGRAM_PLANNING`과 `TODAY_EXECUTION`을 구분했다.
- `FatigueSlotPolicy.allows`의 기본값은 today execution으로 유지하고, ProgramBuilder만 planning 모드를 명시한다.
- planning 모드는 `LOW_LOAD_ONLY` 후보를 제거하지 않고 이후 처방 축소 대상으로 남긴다.

파일별 변경:
- `FatigueSlotPolicy.kt`: fatigue use-case enum과 planning allow path 추가.
- `ProgramBuilder.kt`: schedule adaptation과 slot fatigue filter에서 planning use-case 사용.
- `ProgramBuilderFatigueAndSelectionV041106Test.kt`: today execution hard block 유지와 planning 후보 유지 검증.

동작 변화:
- 미래 프로그램 생성은 fatigue 때문에 foundation anchor를 후보 단계에서 제거하지 않는다.
- 오늘 프로그램 적용 경로의 hard block/downgrade 동작은 유지된다.

다음 작업자 참고:
- planning mode가 후보를 남기더라도 처방은 다음 커밋에서 volume/RPE를 낮춰야 한다.
- execution default를 바꾸면 `ProgramPlanService` 적용 안전 동작이 바뀔 수 있으므로 유지해야 한다.

검증:
- `.\gradlew.bat :app:testDebugUnitTest --tests "*ProgramBuilderFatigueAndSelectionV041106Test*" --tests "*FatigueSlotPolicyTest*"` 통과.

## Commit 3 - fix(program): downscale foundation anchors instead of filtering them

수정 원인:
- planning mode가 foundation anchor 후보를 남겨도 처방과 중량 강도가 그대로면 fatigue 조정 목적이 약해진다.
- 사용자가 원하는 동작은 anchor 제거가 아니라 set/RPE/weight intensity 축소다.

수정 골자:
- heavy lower foundation anchor가 ORANGE/RED planning fatigue에서 2세트 이하로 제한되도록 했다.
- RED heavy lower anchor의 planned RPE cap을 6으로 낮췄다.
- ProgramBuilder weight suggestion intensity multiplier에 planning fatigue load factor를 곱했다.

파일별 변경:
- `ProgramPrescriptionPolicy.kt`: foundation anchor fatigue downscale 규칙 추가.
- `ProgramBuilder.kt`: fatigue-aware weight suggestion intensity 적용.
- `ProgramBuilderFatigueAndSelectionV041106Test.kt`: RED fatigue foundation anchor 처방 축소 검증.

동작 변화:
- future planning에서 anchor 후보는 유지되지만 RED/ORANGE fatigue에서는 작은 dose로 생성된다.
- today execution hard safety gate는 변경하지 않았다.

다음 작업자 참고:
- fatigue downscale 값은 보수적 heuristic이다. 데이터 기반 튜닝이 필요하면 `ProgramPrescriptionPolicy`에서 조정한다.
- 이 커밋은 warning 문구를 바꾸지 않는다. 문구 완화는 다음 커밋에서 처리한다.

검증:
- `.\gradlew.bat :app:testDebugUnitTest --tests "*ProgramBuilderFatigueAndSelectionV041106Test*" --tests "*ProgramPrescriptionPolicyBehaviorTest*"` 통과.

## Commit 4 - fix(program): soften fatigue wording for planning adjustments

수정 원인:
- planning fatigue warning/reason이 사용자에게 운동 금지처럼 읽히면 실제 동작인 "강도 조정"과 맞지 않는다.
- v0.4.1.10.6의 핵심은 foundation anchor 삭제가 아니라 계획상 강도 축소다.

수정 골자:
- ProgramBuilder fatigue warning을 "계획 피로도 참고" 문구로 바꿨다.
- selection reason도 `계획피로조정`으로 표시해 planning adjustment임을 명확히 했다.

파일별 변경:
- `ProgramBuilder.kt`: fatigue warning wording 변경.
- `ProgramSelectionReasonFormatter.kt`: fatigue reason wording 변경.
- `ProgramBuilderFatigueAndSelectionV041106Test.kt`: planning fatigue reason wording 검증.

동작 변화:
- 생성 결과의 안내 문구가 hard block 대신 planning adjustment 의미를 전달한다.
- 후보 선택/처방 로직은 변경하지 않았다.

다음 작업자 참고:
- 오늘 실행 gate 관련 문구와 혼동하지 말 것.
- 상세 fatigue 점수표를 UI에 노출하는 작업은 이번 범위가 아니다.

검증:
- `.\gradlew.bat :app:testDebugUnitTest --tests "*ProgramBuilderFatigueAndSelectionV041106Test*"` 통과.
