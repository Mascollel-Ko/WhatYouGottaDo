
## 4. What Was Already In Place Before This Commit

Previous commit `f260b16` already connected readiness gate data through this path:

```text
TrainingViewModel.applyProgram(...)
-> TrainingRepository.applyProgramToDates(...)
-> FatigueSlotPolicy.adjustTodayItem(...)
-> insert WorkoutEntry / WorkoutSet rows
```

Important existing limitation:

```text
The gate applies only while program items are being inserted by applyProgramToDates(...).
It only adjusts items whose resolved program date equals today.
It does not retroactively rewrite already-inserted planned WorkoutEntry / WorkoutSet rows.
```

This limitation remains intentional after this hardening commit.

## 6. Production Change 1: FatigueSlotPolicy Date Boundary Function

File:

```text
app/src/main/java/com/training/trackplanner/data/FatigueSlotPolicy.kt
```

Added:

```kotlin
fun adjustItemForResolvedDate(
    item: TrainingProgramItem,
    itemDate: String,
    todayDate: String?,
    candidate: ProgramCandidate?,
    gate: ProgramFatigueGate?
): TrainingProgramItem? =
    if (todayDate == null || gate == null || itemDate != todayDate) {
        item
    } else {
        adjustTodayItem(item, candidate, gate)
    }
```

Purpose:

```text
Make the current-day boundary explicit and directly testable.
Keep non-today items unchanged.
Keep null-gate behavior unchanged.
Delegate actual gate decisions to the existing adjustTodayItem(...) logic.
```

Why this exists:

```text
Repository-level Room setup for unit tests was not already present.
Creating a fake database framework just for this would be larger than the behavior under test.
This small function captures the real repository boundary condition and lets the focused unit test verify it.
```

This is a minimal hardening refactor, not a feature expansion.

## 7. Production Change 2: TrainingRepository Delegates Date Boundary

File:

```text
app/src/main/java/com/training/trackplanner/data/TrainingRepository.kt
```

Before this commit, `applyProgramToDates(...)` had the date check inline:

```text
if todayGateContext?.date == itemDate:
    adjustTodayItem(...)
else:
    item
```

After this commit, it delegates to:

```kotlin
val adjustedItem = fatigueSlotPolicy.adjustItemForResolvedDate(
    item = item,
    itemDate = itemDate,
    todayDate = todayGateContext?.date,
    candidate = todayGateContext?.candidatesByExerciseId?.get(item.exerciseId),
    gate = todayGateContext?.gate
) ?: return@forEachIndexed
```

Behavior is intended to remain the same:

```text
null gate -> original item
non-today item -> original item
today item + downgrade gate -> copied item with reduced setCount
today item + avoid gate -> null, so the item is skipped during insertion
```

Stored `TrainingProgramItem` rows are still not mutated.

## 12. Repository-Level Test Decision

A full repository-level Room test was not added.

Reason:

```text
Existing unit tests do not have a TrainingRepository in-memory Room harness.
Existing Room database tests live under androidTest.
Building a new fake TrainingDatabase / DAO framework would be more code than the behavior being hardened.
```

Minimal replacement:

```text
The repository now calls FatigueSlotPolicy.adjustItemForResolvedDate(...).
That function owns the date boundary.
The unit tests cover that exact function directly.
```

This keeps the hardening commit small and avoids adding a fake database framework.

## 14. rpeCap Still Not Wired

This commit intentionally did not wire `rpeCap`.

Reason:

```text
TrainingProgramItem has no planned target RPE field.
WorkoutEntry.rpe and WorkoutSet.rpe are recorded effort fields.
Writing readiness rpeCap into those fields would confuse planned target effort with completed effort.
```

Correct future path:

```text
Add an explicit planned target RPE field to the resolved-session/program-item model first.
Then cap only that planned target field.
Do not write rpeCap into WorkoutEntry.rpe or WorkoutSet.rpe.
```

## 18. Recommended Next Step

Do not implement retroactive planned workout rewriting yet unless explicitly requested.

If that future feature is requested, it should be a separate task with a different boundary:

```text
existing planned WorkoutEntry / WorkoutSet rows
-> only unconfirmed planned rows
-> preserve confirmed user data
-> avoid modifying recorded RPE
-> probably requires repository-level androidTest or an existing Room harness
```

For now, the hardening target is complete:

```text
TrainingGateSnapshot affects only program items being inserted for today's resolved date.
Future program items remain unchanged.
Null gate remains legacy behavior.
Already-inserted planned workout rows are not retroactively modified.
```
