## Commit 1 — test(program): protect builder refactor behavior

Commit:
- `c5c114f`

수정 원인:
- v0.4.1.10.2에서 후보 붕괴를 고쳤지만, 이후 책임 분리 중 adaptive pool, required slot fallback, optional budget trim 완화가 다시 깨질 수 있었다.
- ProgramBuilder 안에는 후보 조회, selection pool 크기, required fallback, session time budget 대응이 아직 함께 남아 있었다.
- 이 책임들은 후보 붕괴와 직접 연결되므로 리팩터링 전에 테스트 보호 장치가 필요했다.

수정 골자:
- `ProgramBuilderCandidateCollapseTest`를 보강했다.
- seed 기반 프로그램이 세션당 1~4개 운동으로 붕괴하지 않는지 계속 확인한다.
- slot query가 fixed top-3가 아니라 adaptive pool을 유지하는지 직접 검증한다.
- required slot이 weak fallback 후보를 사용할 수 있는지 확인한다.
- tight session budget에서 전체 세션을 비우지 않고 trim warning을 남기는지 확인한다.
- 동작 변경 없는 테스트 보강이다.

파일별 변경:
- `app/src/test/java/com/training/trackplanner/data/ProgramBuilderCandidateCollapseTest.kt`: 후보 붕괴, adaptive pool, required fallback, optional budget trim 보호 테스트 추가.
- `docs/v0.4.1.10.3_program_builder_refactor_worklog.md`: v0.4.1.10.3 커밋별 작업 로그 시작.

다음 작업자 참고:
- `.take(3)` 같은 고정 소형 pool을 되살리면 이 테스트가 깨져야 한다.
- required slot exact match가 없더라도 viable weak/fallback 후보가 있으면 unfilled로 떨어지면 안 된다.
- tight budget에서 optional slot을 줄이거나 skip할 수는 있지만, 전체 세션을 비우는 방향으로 돌아가면 안 된다.

## Commit 2 — refactor(program): extract candidate domain model

Commit:
- `743eef0`

수정 원인:
- ProgramBuilder 하단에 후보 모델, 선택 이력, history 기반 중량 추천 상태가 함께 남아 있었다.
- 후보 모델과 선택 이력이 builder 안에 있으면 후보 권한/반복 penalty/coverage accounting을 추적하기 어렵고, 이후 scoring/variety 분리가 더 위험해진다.
- 이 책임은 후보 붕괴 방지와 유지보수성에 연결되지만 사용자 동작을 바꿀 필요는 없었다.

수정 골자:
- `ProgramCandidate.kt`를 추가해 후보 identity, slot fit, role fit, metadata token 해석을 이동했다.
- `ProgramSelectionHistory.kt`를 추가해 stableKey/redundancy/family/slot exposure 상태를 이동했다.
- `ProgramHistoryWeightIndex.kt`를 추가해 history 기반 중량 추천 상태와 결과 모델을 이동했다.
- ProgramBuilder는 기존 객체를 그대로 사용하되 파일 외부 클래스로 참조한다.
- 순수 리팩터링이며 v0.4.1.10.2 후보/중량/반복 penalty 동작을 보존했다.

파일별 변경:
- `ProgramCandidate.kt`: ProgramCandidate와 후보 fit/eligibility helper 책임.
- `ProgramSelectionHistory.kt`: 선택 이력, 반복 penalty, coverage exposure 누적 책임.
- `ProgramHistoryWeightIndex.kt`: confirmed history 기반 weight suggestion 책임.
- `ProgramBuilder.kt`: 이동된 클래스 제거 및 import 정리.
- `docs/v0.4.1.10.3_program_builder_refactor_worklog.md`: Commit 2 기록.

다음 작업자 참고:
- 이 커밋은 정책 변경이 아니라 모델/상태 분리다.
- scoring은 아직 ProgramBuilder에 남아 있으며 다음 커밋에서 `ProgramScoringPolicy`로 이동한다.
- `ProgramCandidate`의 hard eligibility helper와 scoring helper가 아직 섞여 있으므로 scoring extraction 때 hard gate를 옮기지 않도록 주의한다.

## Commit 3 — refactor(program): extract scoring policy

Commit:
- `70eab54`

수정 원인:
- ProgramBuilder에 candidate score 계산, exposure balance, recovery/phase penalty가 남아 있었다.
- 점수 계산이 builder에 남으면 hard gate와 soft preference가 섞여 후보 붕괴 원인을 추적하기 어렵다.
- scoring은 탈락 판단이 아니라 ranking/preference 책임이므로 별도 policy로 분리해야 했다.

수정 골자:
- `ProgramScoringPolicy.kt`를 추가했다.
- `score()`와 `exposureBalanceAdjustment()`를 ProgramBuilder에서 ProgramScoringPolicy로 이동했다.
- performance week / recovery context penalty 상수도 scoring policy 내부로 이동했다.
- hard gate, fatigue gate, session gate, template gate는 ProgramBuilder/Query/기존 policy에 그대로 남겼다.
- 동작 변경 없는 scoring 책임 분리다.

파일별 변경:
- `ProgramScoringPolicy.kt`: candidate ranking 점수와 exposure balance 보정 책임.
- `ProgramBuilder.kt`: score 계산을 `scoringPolicy.score(...)`로 위임하고 scoring helper 제거.
- `docs/v0.4.1.10.3_program_builder_refactor_worklog.md`: Commit 3 기록.

다음 작업자 참고:
- ProgramScoringPolicy는 탈락을 결정하지 않는다. hard safety gate를 여기에 넣지 말 것.
- 후보가 부족하면 query trace/session constraint를 먼저 보고, 점수 policy는 ranking 품질을 볼 것.
- 다음 커밋은 adaptive pool과 controlled selection을 `ProgramVarietyPolicy`로 분리한다.

검증:
- `.\gradlew.bat :app:compileDebugKotlin` 통과.
- `.\gradlew.bat :app:testDebugUnitTest --tests "*ProgramBuilderCandidateCollapseTest*" --tests "*ProgramArchitectureFoundationTest*" --tests "*ProgramBuilderV0352Test*"` 통과.

## Commit 4 — refactor(program): extract variety selection policy

Commit:
- `dca6eb3`

수정 원인:
- ProgramBuilder가 adaptive pool size, controlled candidate selection, repeat allowance, selection history recording을 직접 갖고 있었다.
- 후보 선택/다양성 정책이 builder 내부에 남아 있으면 candidate collapse가 scoring 문제인지, session constraint 문제인지, variety 정책 문제인지 추적하기 어렵다.
- 특히 v0.4.1.10.2에서 제거한 fixed top-3식 후보 압축이 다시 들어오면 전체 distinct exercise가 줄어드는 회귀가 생긴다.

수정 골자:
- `ProgramVarietyPolicy.kt`를 추가했다.
- `selectionPoolSize`, `chooseControlledCandidate`, repeat allowance 위임, selection history recording을 ProgramBuilder에서 ProgramVarietyPolicy로 이동했다.
- adaptive pool 최소 크기와 preference 기반 controlled selection 동작은 기존 v0.4.1.10.2 의미를 유지했다.
- 동작 변경 없는 다양성/선택 정책 분리다.

파일별 변경:
- `ProgramVarietyPolicy.kt`: adaptive pool size, controlled candidate selection, repeat allowance, selection history recording 책임.
- `ProgramBuilder.kt`: candidate selection과 history recording을 `varietyPolicy`로 위임하고 관련 private helper 제거.
- `docs/v0.4.1.10.3_program_builder_refactor_worklog.md`: Commit 4 기록.

다음 작업자 참고:
- ProgramVarietyPolicy는 후보 pool을 넓게 유지하고 반복을 조절하는 책임이다. hard safety gate를 이 파일로 옮기지 말 것.
- `.take(3)` 같은 고정 소형 pool을 되살리면 v0.4.1.10.2 candidate collapse 회귀가 난다.
- 다음 커밋은 heavy lower, high impact, COD, neural stress 같은 session safety 조합 제한을 `ProgramSessionConstraintPolicy`로 분리한다.

검증:

- `ProgramSelectionReasonFormatter.kt`를 추가하고 `reasonTokens()` 문자열 구성을 이동했다.
- ProgramBuilder는 중량 추천과 reason formatting을 각각 policy/formatter에 위임한다.
- 동작 변경 없는 helper 책임 분리다.

파일별 변경:
- `ProgramWeightSuggestionPolicy.kt`: confirmed history 기반 weight suggestion 책임.
- `ProgramSelectionReasonFormatter.kt`: selection reason 문자열 구성 책임.
- `ProgramHistoryWeightIndex.kt`: 새 weight policy로 이동되어 삭제.
- `ProgramBuilder.kt`: weight suggestion과 reason formatting을 위임하고 private helper 제거.
- `docs/v0.4.1.10.3_program_builder_refactor_worklog.md`: Commit 6 기록.

다음 작업자 참고:

- ProgramSelectionReasonFormatter는 판단하지 않고 표시 문자열만 만든다.
