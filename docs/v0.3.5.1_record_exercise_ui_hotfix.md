# v0.3.5.1 Record and Exercise UI Hotfix

## Scope

- Version remains `v0.3.5.1` / `305001`.
- Record numeric fields clear a stored zero only while focused and restore zero when an empty field loses focus.
- Today's exercise order follows the first-confirmation sequence and is persisted in Room.
- A newly added exercise is inserted after the latest confirmed exercise and the Record list scrolls to the insertion context.
- Strength and functional exercise lists expose metadata-based subcategories with search-filtered counts.
- Calendar/Record view behavior is unchanged; the hotfix does not force Calendar as the default.

## Record Ordering

`WorkoutEntry.displayOrder` stores the visible order and `WorkoutEntry.firstConfirmedAt` records the first confirmation time. On the first confirmed set, the entry is moved after the previously confirmed entry. Later set edits do not move it again.

Example: planned `A, B, C`, confirmed `A, C, B` results in persisted order `A, C, B`. Adding `D` after `C` inserts it as `A, C, D, B`.

Database migration `13 -> 14` adds both columns without destructive migration. Existing rows are ordered by their prior IDs, and rows with confirmed sets receive a compatible first-confirmation value.

## Exercise Subcategories

Subcategory assignment uses runtime canonical metadata and structured exercise fields. Exercise names are used only for user-entered search filtering, not for category inference. Strength and functional groups show only non-empty subcategories, and counts are recalculated from the current search result.
