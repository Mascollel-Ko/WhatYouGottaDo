## 2. Why This Change Exists

The clean `v0.3.6.4` readiness pipeline already had the right high-level structure:

```text
DailyAnalysisLoadAggregator
-> ResidualFatigueCalculator
-> StatisticalBaselineCalculator
-> AdaptiveBaselineCalculator
-> FatiguePressureCalculator
-> TodayReadinessDecisionEngine
```

The specific weakness fixed here was inside `ResidualFatigueCalculator`.

Before this change:

```kotlin
private fun categoryAdjustedDecay(category: FatigueCategoryKey, baseDecay: Double): Double =
    when (category) {
        FatigueCategoryKey.NEURAL_HEAVY,
        FatigueCategoryKey.DECELERATION,
        FatigueCategoryKey.ELASTIC_SSC -> baseDecay.coerceAtLeast(baseDecay * 1.05)
        else -> baseDecay
    }
```

Problems:

```text
1. Same-day decay could exceed 1.0 because 1.00 * 1.05 = 1.05.
2. Residual fatigue could become larger than the original load.
3. Category persistence was only a shallow multiplier.
4. RPE was already known by the analysis mapper, but it was not carried into residual fatigue.
5. Future-dated contributions were skipped, but decay factor clamping was not explicit.
```

The new implementation fixes those problems while staying inside the existing readiness pipeline.

### 3.3 `ResidualFatigueCalculator.kt`

Path:

```text
app/src/main/java/com/training/trackplanner/analysis/readiness/ResidualFatigueCalculator.kt
```

Key changes:

```text
1. Added explicit recovery profile order.
2. Added profile normalization.
3. Clamped all decay factors to 0.0..1.0.
4. Replaced multiplier-based category adjustment with profile promotion.
5. Added RPE-aware persistence promotion.
6. Kept body-part and baseline-group residuals on the base decay profile for now.
```

Current profile order:

```kotlin
private val profileOrder = listOf("MINIMAL", "SHORT", "MEDIUM", "LONG", "VERY_LONG")
```

Current public decay function:

```kotlin
fun decayFactor(profile: String, ageDays: Int): Double {
    if (ageDays < 0) return 0.0
    val curve = TodayReadinessConstants.decayCurves[normalizeProfile(profile)]
        ?: TodayReadinessConstants.decayCurves.getValue("SHORT")
    return curve.getOrElse(ageDays) { 0.0 }.coerceIn(0.0, 1.0)
}
```

Important behavior:

```text
ageDays < 0 returns 0.0.
Unknown or blank profile falls back to SHORT.
Every returned decay factor is clamped to 0.0..1.0.
```

Category residuals now use category-specific adjusted decay:

```kotlin
contribution.categoryLoads.forEach { (category, load) ->
    val decay = categoryAdjustedDecay(
        category = category,
        profile = contribution.recoveryDecayProfile,
        ageDays = ageDays,
        averageRpe = contribution.averageRpe
    )
    categoryResiduals.add(category, load * decay)
}
```

Body-part and adaptive baseline group residuals still use the base profile:

```kotlin
val baseDecay = decayFactor(contribution.recoveryDecayProfile, ageDays)
if (baseDecay > 0.0) {
    contribution.bodyPartLoads.forEach { (part, load) ->
        bodyPartResiduals.add(part, load * baseDecay)
    }
    contribution.baselineGroupLoads.forEach { (group, load) ->
        baselineGroupResiduals.add(group, load * baseDecay)
    }
}
```

This was deliberate. The first improvement only changes category persistence, because `FatiguePressureSnapshot.categoryPressures` is the canonical fatigue category pressure source. Body-part and baseline group persistence can be revisited later when tissue/body-part gates are integrated.

## 4. Residual Persistence Taxonomy

This is the most important part for the next developer or AI.

The canonical internal fatigue taxonomy remains:

```kotlin
enum class FatigueCategoryKey {
    SYSTEMIC,
    NEURAL_HEAVY,
    NEURAL_SPEED,
    LOCAL_MUSCLE,
    DECELERATION,
    ELASTIC_SSC,
    ROTATION_POWER,
    ANTI_ROTATION,
    OVERHEAD_REPETITION,
    GRIP_FOREARM,
    BADMINTON_COURT
}
```

No new taxonomy was created.

The new taxonomy-like behavior is a persistence layer on top of `FatigueCategoryKey`.
It answers this question:

```text
Given a category, base recovery profile, age in days, and average RPE,
which existing decay curve should be used for residual category fatigue?
```

It does not change category load creation.
It does not change metadata classification.
It does not change UI labels.
It only changes residual persistence.

### 4.1 Base Decay Profiles

Existing decay curves in `TodayReadinessConstants`:

```kotlin
val decayCurves: Map<String, List<Double>> = mapOf(
    "MINIMAL" to listOf(1.00, 0.15),
    "SHORT" to listOf(1.00, 0.50, 0.20),
    "MEDIUM" to listOf(1.00, 0.65, 0.40, 0.20),
    "LONG" to listOf(1.00, 0.80, 0.55, 0.35, 0.20),
    "VERY_LONG" to listOf(1.00, 0.90, 0.70, 0.50, 0.35, 0.20)
)
```

Profile rank:

```text
MINIMAL < SHORT < MEDIUM < LONG < VERY_LONG
```

Promotion means moving right in that order.

Examples:

```text
MINIMAL + 1 step = SHORT
SHORT + 1 step = MEDIUM
MEDIUM + 1 step = LONG
LONG + 1 step = VERY_LONG
VERY_LONG + any step = VERY_LONG
```

### 4.2 Category Persistence Rules

Current category persistence defaults:

| Category | Default persistence adjustment | Reasoning |
|---|---:|---|
| `SYSTEMIC` | 0 steps | Uses metadata/profile as-is for now. |
| `NEURAL_HEAVY` | +1 step | Heavy neural fatigue should persist longer than ordinary local muscle fatigue. |
| `NEURAL_SPEED` | 0 steps | Speed work only gets persistence promotion when RPE is high. |
| `LOCAL_MUSCLE` | 0 steps | Baseline comparison category for ordinary local muscular fatigue. |
| `DECELERATION` | +1 step | Braking/change-of-direction stress should persist longer than ordinary local fatigue. |
| `ELASTIC_SSC` | +1 step | Jump/landing/stretch-shortening stress should persist longer than ordinary local fatigue. |
| `ROTATION_POWER` | 0 steps | No special residual rule yet. |
| `ANTI_ROTATION` | 0 steps | No special residual rule yet. |
| `OVERHEAD_REPETITION` | 0 steps | No special residual rule yet. |
| `GRIP_FOREARM` | 0 steps | No special residual rule yet. |
| `BADMINTON_COURT` | 0 steps | Composite court load is not promoted yet; its components carry persistence. |

Current code:

```kotlin
val categorySteps = when (category) {
    FatigueCategoryKey.NEURAL_HEAVY,
    FatigueCategoryKey.DECELERATION,
    FatigueCategoryKey.ELASTIC_SSC -> 1
    else -> 0
}
```

These are beta heuristics, not research-derived constants.

### 4.3 RPE Persistence Rules

Current RPE thresholds:

```text
RPE is null: no RPE persistence promotion.
RPE < 8.0: no RPE persistence promotion.
RPE >= 8.0: selected categories get +1 additional profile step.
RPE >= 9.0: selected high-impact/high-neural categories are promoted to at least LONG.
```

RPE `>= 8.0` category behavior:

| Category | RPE >= 8.0 adjustment |
|---|---:|
| `NEURAL_HEAVY` | +1 step |
| `NEURAL_SPEED` | +1 step |
| `DECELERATION` | +1 step |
| `ELASTIC_SSC` | +1 step |
| All other categories | 0 steps |

Current code:

```kotlin
val rpeSteps = when {
    averageRpe == null || averageRpe < 8.0 -> 0
    category == FatigueCategoryKey.NEURAL_SPEED -> 1
    category in highRpePersistenceCategories -> 1
    else -> 0
}
```

RPE `>= 9.0` minimum-profile rule:

| Category | RPE >= 9.0 minimum profile |
|---|---|
| `NEURAL_HEAVY` | `LONG` |
| `DECELERATION` | `LONG` |
| `ELASTIC_SSC` | `LONG` |
| `NEURAL_SPEED` | No minimum; only the RPE >= 8.0 +1 step applies. |
| All other categories | No minimum. |

### 4.4 Final Adjusted Profile Algorithm

Current algorithm:

```text
1. Normalize base profile.
2. Compute categorySteps.
3. Compute rpeSteps.
4. Promote base profile by categorySteps + rpeSteps.
5. If RPE >= 9.0 and category is NEURAL_HEAVY, DECELERATION, or ELASTIC_SSC, require at least LONG.
6. Pick the slower profile between the promoted profile and the RPE-9 minimum profile.
7. Compute base decay and adjusted-profile decay.
8. Use max(base decay, adjusted decay).
9. Clamp final decay to 0.0..1.0.
```

Current code:

```kotlin
private fun categoryAdjustedDecay(
    category: FatigueCategoryKey,
    profile: String,
    ageDays: Int,
    averageRpe: Double?
): Double {
    val baseProfile = normalizeProfile(profile)
    val adjustedProfile = categoryAdjustedProfile(category, baseProfile, averageRpe)
    return maxOf(
        decayFactor(baseProfile, ageDays),
        decayFactor(adjustedProfile, ageDays)
    ).coerceIn(0.0, 1.0)
}
```

```kotlin
private fun categoryAdjustedProfile(
    category: FatigueCategoryKey,
    baseProfile: String,
    averageRpe: Double?
): String {
    val categorySteps = when (category) {
        FatigueCategoryKey.NEURAL_HEAVY,
        FatigueCategoryKey.DECELERATION,
        FatigueCategoryKey.ELASTIC_SSC -> 1
        else -> 0
    }
    val rpeSteps = when {
        averageRpe == null || averageRpe < 8.0 -> 0
        category == FatigueCategoryKey.NEURAL_SPEED -> 1
        category in highRpePersistenceCategories -> 1
        else -> 0
    }
    val rpeNineMinimum = if (averageRpe != null && averageRpe >= 9.0 && category in highRpePersistenceCategories) {
        "LONG"
    } else {
        baseProfile
    }
    return maxProfile(
        promoteProfile(baseProfile, categorySteps + rpeSteps),
        rpeNineMinimum
    )
}
```

### 4.6 Same-Day Behavior

All existing decay curves start at:

```text
ageDays = 0 -> 1.00
```

Because decay factors are clamped:

```text
same-day residual = original load * 1.00
```

Same-day residual category load must not exceed the original category load.

This is the main safety fix compared with the old `baseDecay * 1.05` behavior.

### 4.7 Future-Dated Records

Future-dated records have:

```text
ageDays < 0
```

They are ignored in `calculate(...)`:

```kotlin
if (ageDays < 0) return@forEach
```

And `decayFactor(...)` also returns zero for negative ages:

```kotlin
if (ageDays < 0) return 0.0
```

This gives two layers of protection.

## 8. Guidance For The Next Developer Or AI

If continuing the v0.3.9.0 fatigue engine work, keep these rules:

```text
1. Reuse FatigueCategoryKey.
2. Do not add another fatigue taxonomy unless a concrete gap is proven.
3. Treat ResidualFatigueCalculator as the residual-load source for the readiness pipeline.
4. Treat FatiguePressureCalculator as the pressure source for Today Readiness, future fatigue UI, and future ProgramBuilder gates.
5. Keep RPE persistence effects on decay/persistence, not same-day magnitude amplification.
6. Preserve the 0.0..1.0 decay clamp.
7. Keep future-dated records contributing zero residual load.
8. Add tests before expanding persistence behavior to body parts or tissue stress.
```

Recommended next steps:

```text
1. Add or update the audit doc for v0.3.9.0 if not already done.
2. Decide whether bodyPartResiduals should later receive tissue-aware persistence.
3. Build a shared fatigue presentation/gate snapshot from FatiguePressureSnapshot.
4. Rewire FatigueSlotPolicy only after the shared gate snapshot exists.
5. Rebuild analysis/fatigue only after the readiness pressure snapshot is the confirmed source.
```

Do not do next:

```text
Do not expand analysis/fatigue as an independent calculator.
Do not add FatigueDomain.
Do not add FatigueVectorResolver.
Do not mutate metadata rows for this residual decay work.
Do not bump version until the broader v0.3.9.0 acceptance criteria are met.
```
