## Commit 2 - fix(program): preserve loaded strength candidates when all equipment is available

수정 원인:
- The reproduction test showed that all-equipment badminton support plans could still produce no loaded strength items.
- The seed fixture stores equipment tags in Korean tokens such as `바벨`, `덤벨`, `머신`, and `케이블`, while the loaded-strength recognition path only checked English equipment tokens.
- This made normal barbell/dumbbell/machine/cable strength work invisible to the loaded-strength scoring path even when all equipment was selected.

수정 골자:
- Added a loaded-strength flag to `ProgramCandidate`.
- Added a small loaded-strength scoring boost in `ProgramScoringPolicy` when loaded equipment is available.
- Treated both English and Korean seed equipment tokens as loaded strength equipment.
- Enabled the reproduction test that was intentionally ignored in Commit 1.
- This keeps the equipment hard gate intact; it does not allow unselected equipment.

파일별 변경:
- `ProgramCandidate.kt`: identifies loaded strength candidates from exercise equipment and strength-fit metadata.
- `ProgramScoringPolicy.kt`: gives loaded strength candidates a bounded preference so they stay in the selection pool when equipment is available.
- `ProgramBuilderQualityV041104Test.kt`: enables the all-equipment loaded-strength regression and accepts Korean seed equipment tokens.

동작 변화:
- All-equipment selection now preserves ordinary loaded strength candidates instead of letting bodyweight/small-tool/transfer candidates dominate every slot.
- The existing selected-equipment hard gate remains unchanged.
- 6-corner shadow footwork remains allowed.

다음 작업자 참고:
- This commit only fixes loaded-strength recognition and ranking.
- It does not enforce the 60:40 composition quota; that belongs in the next composition-focused commit.
- Do not remove Korean equipment tokens unless the seed/import path is normalized first.

검증:

동작 변화:

다음 작업자 참고:

## Commit 8 - feat(program): optimize generated skeleton with evaluation repair loop

수정 원인:
- Evaluation issues needed a bounded automatic loop so severe fatigue clustering could be repaired before returning the final skeleton.
- The repair loop must improve quality only when measurable and must not violate equipment, exclusion, or time constraints.

수정 골자:
- Added optimization trace and summary models.
- Added `ProgramRepairPolicy` for a conservative adjacent high-lower fatigue repair.
- Added `ProgramOptimizationPolicy` with a maximum of three iterations and accept-only-if-improved behavior.
- Connected ProgramBuilder to optimization after validation.
- Added optimization tests for accepted repair, bounded iteration count, summary output, and no-op healthy skeleton behavior.

파일별 변경:
- `ProgramOptimizationTrace.kt`: defines optimization summary, trace, and repair result models.
- `ProgramRepairPolicy.kt`: softens the later day in an adjacent high-lower/COD fatigue cluster.
- `ProgramOptimizationPolicy.kt`: runs evaluate -> repair -> re-evaluate loop with bounded acceptance.
- `ProgramSkeletonGenerator.kt`: stores optional evaluation, optimization summary, and trace on generated skeletons.
- `ProgramBuilder.kt`: returns the optimized validated skeleton.
- `ProgramOptimizationPolicyTest.kt`: verifies improvement, bounded loop, accepted trace, summary, and no-added-exercise behavior.

동작 변화:
- Generated skeletons now carry internal evaluation/optimization trace data.
- A conservative repair can lower adjacent high-lower fatigue if the evaluation score improves.
- The loop does not add exercises, does not run indefinitely, and keeps the original skeleton if repair does not improve evaluation.

다음 작업자 참고:
- This loop is intentionally small; future exercise-swap repair needs candidate inventory access and stricter equipment/exclusion checks.
- Do not use optimization to flatten every day into moderate intensity.
- UI should show only the short summary, not the full evaluation score table.

검증:
