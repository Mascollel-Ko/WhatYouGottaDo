## Commit 7 - fix(program): rotate core and accessory patterns before repeating fillers

Cause:
- Core/accessory/filler repetition was still handled inside generic reranking.
- Captain-chair style trunk-flexion work needed a named policy so repeated fillers are penalized before they behave like anchors.

Change Summary:
- Added `ProgramCorePatternPolicy`.
- Moved core-pattern repetition adjustment out of `ProgramCandidateRerankingPolicy`.
- Added builder warnings for repeated trunk-flexion and program-wide core accessory stableKey overuse.

Files:
- `ProgramCorePatternPolicy.kt`: owns core/accessory pattern detection, selection adjustment, and warnings.
- `ProgramCandidateRerankingPolicy.kt`: delegates core-pattern adjustment to the new policy.
- `ProgramBuilder.kt`: includes core/accessory repetition warnings in generated skeleton warnings.
- `ProgramBuilderReservoirBeamV041105Test.kt`: covers captain-chair penalty, anti-rotation rotation, and repetition warnings.

Behavior:
- Repeated trunk-flexion/core filler candidates are penalized.
- Anti-rotation, anti-extension, lateral stability, and carry candidates are preferred after trunk flexion already appears.
- Foundation anchor selection is not blocked by accessory repetition caps.

Next Worker Notes:
- Keep this as a soft scoring policy. Do not turn core rotation into a hard filter except for same-session duplicates or clear safety conflicts.

## Commit 3 - refactor(program): keep eligible candidates in a reservoir

수정 원인:
- The old filter cascade made it too easy for non-hard-gate policies to remove useful candidates before slot-level reranking.
- Foundation, loaded support, transfer, and core/accessory candidates need stable classification data that later policies can consume.

수정 골자:
- Added candidate tier, foundation pattern, and core pattern classification.
- Added `ProgramCandidateReservoir` to preserve hard-gate-eligible candidates and their classifications.
- Extended `ProgramCandidateInventoryResult` with a reservoir while keeping the existing `candidates` list for compatibility.

파일별 변경:
- `ProgramCandidateClassificationPolicy.kt`: classifies candidates into foundation, loaded support, transfer, core/accessory/prehab, or filler tiers.
- `ProgramCandidateReservoir.kt`: stores eligible candidates and classification lookup.
- `ProgramCandidateInventory.kt`: returns the reservoir with inventory results.
- `ProgramBuilderReservoirBeamV041105Test.kt`: verifies foundation and captain-chair/core classification.

동작 변화:
- No selection behavior change yet.
- Hard-gate-eligible candidates now have reservoir/classification metadata for later reranking and repair.

다음 작업자 참고:
- Repeat, fatigue, and variety should use this reservoir as scoring context, not as early destructive filters.
