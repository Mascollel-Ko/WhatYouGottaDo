# v0.4.1.10.5 ProgramBuilder Reservoir / Beam Selection Release Notes

## Baseline

- Started from: latest `main` containing `v0.4.1.10.4`
- Branch: `refactor/program-builder-reservoir-beam-v041105`
- Release version: `versionName = "v0.4.1.10.5"`, `versionCode = 401015`

## Reproduction Scenario

- Goal: badminton support strength training
- Duration: 4 weeks
- Weekly days: 4 or 5
- Daily minutes: 45
- Badminton:strength ratio: 60:40
- Periodization: AUTO or BADMINTON_WAVE
- Equipment: all available equipment selected

## Existing Problems

- Foundation anchors such as squat, hinge, pull, and press could be absent while core/accessory fillers repeated.
- Captain-chair style trunk-flexion work could behave like a safe repeated filler instead of rotating with other core patterns.
- Week 1 through Week 4 could look too similar even when periodization input existed.
- The evaluation model could still allow overly high scores for skeletons with missing anchors or repeated week profiles.
- Progress text still described the older linear generation flow instead of reservoir/beam optimization.

## Why Reservoir / Beam

The prior flow still behaved like a filter cascade: each policy narrowed candidates before final selection. That made safe filler exercises survive while foundation candidates were lost. v0.4.1.10.5 keeps hard-gate-eligible candidates in a reservoir, classifies them, reserves foundation slots, reranks with current program context, and chooses from a bounded top-k/beam window.

## File Responsibility Map

- `ProgramBuilder`: orchestrates generation, candidate inventory, periodization context, slot filling, warnings, validation, and optimization.
- `ProgramPeriodizationPlanPolicy`: derives week roles/day profiles from existing periodization input.
- `ProgramPeriodizationModels`: week-role and day-profile models.
- `ProgramCandidateReservoir`: preserves hard-gate-eligible candidates with classification lookup.
- `ProgramCandidateClassificationPolicy`: classifies candidate tier, foundation pattern, core pattern, movement/equipment family, and transfer goal.
- `ProgramFoundationAnchorPolicy`: reserves foundation anchor slots before accessory/filler slots.
- `ProgramCandidateScoreContext`: passes current program state into reranking.
- `ProgramCandidateRerankingPolicy`: applies soft context-aware score adjustments.
- `ProgramBeamSelectionPolicy`: keeps a bounded top-k slot window and chooses from beam candidates.
- `ProgramCorePatternPolicy`: rotates core/accessory patterns and emits repeated-filler warnings.
- `ProgramIssueDrivenRerankPolicy`: reopens weak slots for issue-driven reservoir replacements.
- `ProgramEvaluationPolicy`: evaluates weekly/program-wide quality, issues, and score caps.
- `ProgramOptimizationPolicy`: runs bounded repair/re-evaluation loops.
- `ProgramRepairPolicy`: delegates issue-driven repair and preserves existing high-lower fatigue softening.
- `TrainingViewModel`: exposes reservoir optimization progress messages and timeout failure.

## Behavior After Fix

- Periodization input now provides week-role/day-profile context before selection.
- Hard-gate-eligible candidates remain in the reservoir instead of being filtered out by soft preferences.
- Foundation anchors are reserved before accessory/core slots in compact sessions.
- Loaded strength candidates can be reranked up when underused.
- Slot selection considers a wider top-k window rather than a fixed top 1-3.
- Core/accessory fillers rotate before repeating captain-chair style trunk flexion.
- Issue-driven repair can reopen weak slots for missing anchors or repeated core patterns.
- Bad skeletons with missing anchors, repeated fillers, sparse density, or identical weeks cannot score 100.
- 6-corner shadow footwork remains allowed as badminton transfer work.
- Detailed evaluation remains internal; the UI shows progress and short optimization summaries only.

## Remaining Risks

- Beam selection is intentionally bounded, so it improves candidate breadth without exhaustive global optimization.
- Issue-driven repair currently targets missing anchors and repeated core patterns. More repair types should be added only with focused tests.
- Progress percentages are stage-based, not actual per-candidate computation progress.
