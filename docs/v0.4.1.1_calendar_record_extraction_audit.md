## Behavior That Must Be Preserved

### Confirmed / Unconfirmed

- `copyDate(..., keepConfirmed=true)` preserves each source set's confirmed state.
- `copyDate(..., keepConfirmed=false)` copies all sets as unconfirmed.
- `copyDateRangeAsPlan` defaults to `keepConfirmed=false`; state-preserving range copy must pass `keepConfirmed=true`.
- `deleteDateRange(includeConfirmed=false)` removes only unconfirmed sets and keeps confirmed sets.
- `ProgramPlanService.applyProgramToDates` creates only unconfirmed planned sets.

### Overwrite / Append

- Calendar copy/move/range-copy use `CalendarConflictMode`.
- Calendar overwrite deletes all destination sets and entries for target dates.
- Program apply uses `ProgramApplyMode`; overwrite deletes planned-only sets/entries between program start/end and preserves confirmed records.
- These overwrite policies are intentionally different.

### Completed Timestamp

- Calendar copy regenerates `completedAt` and `firstConfirmedAt` with current time only when `keepConfirmed=true` and the copied entry has any confirmed set.
- Calendar copy sets both fields to null when state is not preserved.
- `RecordMutationService.updateSet` owns normal completion refresh for single-set updates.
- Program apply inserts planned entries without completion timestamps.

### Display Order / Set Order

- Calendar copy sets copied entry `displayOrder` to source iteration order for the copied batch.
- Calendar copy preserves set order by sorting source sets by `setIndex`.
- `deleteDateRange(includeConfirmed=false)` reindexes remaining set indices after deleting unconfirmed sets.
- `RecordMutationService` owns single-day display-order normalization for add/update single-record flows.

### Date / LocalDate Handling

- `dateRange` parses both endpoints as ISO local dates, uses min/max order, and returns inclusive dates.
- `copyDateRangeAsPlan` maps source date offsets to target dates with `LocalDate.plusDays`.
- Program apply has its own program-item date calculation based on week/day offsets.

### Backup Compatibility

- Calendar copy/delete does not change backup format.
- Restore/import has separate duplicate detection, id/date/order reconstruction, exercise fallback, daily metric/check-in merge, and metadata override restore.
- Do not reuse calendar copy/delete helpers inside restore/import.

## Low-risk Extraction Scope

- Create `CalendarRecordService` with only:
  - `calendarConflictSummary`
  - `deleteDate`
  - `deleteDateRange`
  - `copyDate`
  - `moveDate`
  - `copyDateRangeAsPlan`
  - private `copyEntriesToDate`, `dateRange`, `nextCreatedAt`
- Keep `TrainingRepository` public APIs unchanged and delegate to the service.
- Give the service `TrainingDatabase` and `WorkoutDao`; do not pass `TrainingRepository`.
- Do not touch `RecordMutationService`, `ProgramPlanService`, backup/import/export, UI, or DAO queries.

## Medium-risk Scope

- Moving `CalendarConflictSummary` / `CalendarConflictMode` model definitions can be done later, but it creates more API churn than value.
- Sharing planned-only overwrite code between `ProgramPlanService` and calendar copy is not recommended yet because policies differ.
- Extracting `noteFromPrescription` is only relevant if program/apply helpers are revisited; it is not calendar copy logic.

## High-risk / Do Not Touch Yet

- `ProgramPlanService.applyProgramToDates`: planned-only overwrite and fatigue slot adjustment make it a separate responsibility.
- Restore/import internals: data-loss boundary with legacy compatibility and duplicate detection.
- `RecordMutationService` internals: single-entry/set completion and display-order normalization should remain isolated.
- Batch edit UI/helper logic: value-level set editing, not date-level calendar operations.
- DAO query changes or Room schema changes.
