## Pre-push Refinement

- Direct sport sessions are excluded even if malformed metadata marks them selectable. Court footwork, reaction drills, and repeated-effort tests remain eligible according to their own planning metadata.
- Repetition scoring now treats stable anchors separately from accessory, prehab, and transfer rotation.
- Same `stableKey` receives the strongest penalty, `redundancyGroup` receives a soft recency penalty, and weekly `movementFamily` exposure receives a soft budget penalty.
- Session size is capped by prescription-derived work time, inter-set rest, setup time, and a warm-up reserve.
- Validation uses `HARD`, `WARNING`, and `SOFT_PENALTY` severities.
- Added weekly hard-day rules, two-week neuromuscular/joint-impact wave checks, two-week movement coverage, and four-week distribution checks.
- `appCueProfile` remains a candidate/display helper and is not used as a fatigue or analysis axis.

## Compatibility Decisions

- No Room migration: `durationDays` already stores the selected duration.
- Existing program rows remain readable.
- Slot and day-intensity tokens are stored in the existing prescription text.
- `appCueProfile` is candidate-selection metadata only and is never aggregated into analysis.
- Exercise names are display values and explicit exclusion inputs, not classification inputs.

## Rehab-Like Activation Refinement

- Classification is metadata-first. It combines low-load prehab/control stress, recovery/care slot, quality-based progression, absent strength progression, short/low stress, low-load equipment/rest, and movement subtype signals.
- Loaded strength/compound progression and performance scapular retraction/rear-delt metadata subtract from the heuristic score.
- Canonical results: wall slide, scap push-up, and band external rotation are rehab-like activation. Scap pull-up remains a progressed vertical-pull item. Face pull, Pallof press, and side plank remain performance-supporting alternatives.
- Normal sessions permit at most one rehab-like activation candidate and never allow one in an anchor position.
- The same rehab-like stable key receives a strong 14-day and moderate 28-day repetition penalty.
- Deload, final-deload, RED fatigue, and explicit recovery slots reduce the score penalty and permit up to three candidates, while still retaining exact-item repetition control.
- Pain/injury-specific allowance is not applied because `ProgramSkeletonRequest` does not currently carry joint-specific constraints. No separate onboarding or schema field was invented.
- Validation now detects normal-session over-cap/main-position/dominance, two-week stable-key repetition, four-week share above 15%, and scapular exposure dominated by low-load activation.
