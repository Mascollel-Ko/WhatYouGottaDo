# v0.3.5.0 App Integration Readiness Audit

## Executive Answer

- The app already uses structured Exercise metadata in analysis, fatigue/readiness, badminton recommendations, and program generation.
- The **new final sidecar taxonomy itself is not integrated**. This is an expected integration gap, not a current defect.
- Existing metadata is embedded in the `Exercise` Room entity; there is no separate `ExerciseMetadata` table.
- Records join to metadata through local `exerciseId`. `stableKey` is stored on Exercise, not WorkoutEntry/WorkoutSet.
- Backup exercise rows preserve stableKey, but set rows preserve only exerciseName, so restore identity is only partially stable.

## Current Record-to-Metadata Gap

1. Canonical metadata lives on Exercise and analysis resolves it by WorkoutEntry.exerciseId.
2. WorkoutEntry also stores exerciseName as a snapshot, but does not store stableKey.
3. Backup exports stableKey for exercise catalog rows, not set rows.
4. Restore imports exercise definitions first, then resolves set rows by exerciseName.
5. A rename, missing exercise catalog row, or ambiguous imported name can therefore break canonical identity.
6. Follow-up integration after v0.3.5.0 should introduce one stableKey-first resolver and include stableKey in record backup rows.

## Analysis Interpretation

Current analysis is already metadata-based through ExerciseAnalysisMapper and AnalysisInputCollector. Remaining string operations are primarily seed/import fallbacks, UI search, user exclusion search, and substring mapping of structured muscle tokens. The new taxonomy should extend and tighten this path rather than replace the analysis stack.
