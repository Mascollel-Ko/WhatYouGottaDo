## Selection Rules

- Candidate role comes from canonical `movementFamily`, `programSlot`, `redundancyGroup`, `progressMetricType`, `strengthProgressionGroup`, transfer, and stress metadata.
- Legacy movement fields are used only when canonical runtime metadata cannot resolve a row.
- Exercise names are display values and explicit user-exclusion matches, not role classifiers.
- `RANDOM_BEEP_CUE` boosts eligible footwork/reaction candidates only. It is not added to analysis aggregation.
- COD transfer positions require footwork/reaction/deceleration/app-cue metadata.
- Power/reactive positions require plyometric/elastic/power/reaction metadata.
- Stable anchors may repeat for progression continuity. Non-anchor stable keys receive a strong 7-day and mild 14-day repetition penalty.
- Recent `redundancyGroup` exposure and weekly `movementFamily` concentration receive soft scoring penalties.
- Direct sport sessions, matches, lessons, and sport-session placeholders cannot enter generated gym/support programs.
- Session duration is estimated from prescriptions, rest, setup, and warm-up reserve before an item is accepted.

## Validation Layers

| Severity | Use |
|---|---|
| HARD | direct sport session leakage, hard-day rule violation, RED hard day, session time overflow |
| WARNING | two-week neuromuscular or joint-impact wave, movement coverage, four-week transfer/stress/rotation debt |
| SOFT_PENALTY | movement-family concentration, redundancy exposure, informational axis/distribution debt |

Weekly hard-day limits are 1 day for 1-4 day schedules and 2 days for normal 5-7 day schedules. A HIGH/INTENSIFY week may use 3 only when current fatigue is not ORANGE/RED and a low-load/deload week follows. RED always permits 0 hard days.

Two-week coverage checks squat/single-leg, hinge, upper pull, trunk, scapular, footwork/COD, and calf/ankle exposure. Four-week checks cover transfer groups, strength progression groups, movement families, stress distribution, accessory rotation, and anchor continuity.

## Rehab-Like Activation Cap

The cap does not blacklist exercise names. The classifier uses canonical `movementFamily`, `movementSubtype`, `programSlot`, `redundancyGroup`, `progressMetricType`, `strengthProgressionGroup`, `primaryStressProfile`, secondary stress tags, `stressMagnitudeHint`, `recoveryDurationClass`, equipment, rest time, and transfer/progression context.

Positive signals include low-load prehab/control stress, recovery or rotator-cuff-care slots, quality-based non-progressive work, short low stress, and bodyweight/band/wall low-rest execution. Loaded strength/compound progression and performance scapular retraction/rear-delt accessory metadata reduce the score.

| Canonical example | Handling |
|---|---|
| Wall slide | rehab-like activation; normal-session cap and repeat penalty |
| Scap push-up | rehab-like activation; normal-session cap and repeat penalty |
| Band external rotation | rehab-like activation; allowed occasionally, not every normal week |
| Scap pull-up | progressed vertical-pull/scapular strength; not rehab-like |
| Face pull | performance-supporting scapular accessory; not rehab-like |
| Pallof press / side plank | trunk stability alternatives; not rehab-like |

Normal BUILD/HIGH/INTENSIFY/performance sessions allow at most one rehab-like item, apply a stronger score penalty, and prohibit it from anchor position. Deload/final-deload, RED fatigue, and recovery/prehab slots reduce the penalty and allow up to three while preserving exact stable-key repetition control. The same rehab-like stable key receives a strong 14-day penalty.

The validator warns on multiple normal-session items, main-position use, two-week repeat, four-week share above 15%, and scapular exposure dominated by activation. A normal strength/transfer session that is at least half rehab-like with two or more such items is a HARD issue.

Joint-specific pain/injury allowance is deferred because the current builder request has no pain/constraint input. The implementation does not infer injury from exercise names or create a parallel profile model.
