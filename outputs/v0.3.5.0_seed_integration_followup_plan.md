# v0.3.5.0 Seed Integration Follow-up Plan

## Constraint

Do not copy the Pass 3 sidecar over `training_settings_seed.csv` or `exercises_seed.json`. The runtime adapter is ready, but canonical metadata still needs a reviewed delivery and persistence design.

## Recommended Sequence

1. Generate a report-only runtime resource candidate containing only canonical production fields and identity fields.
2. Validate 215 rows, unique stable keys, approved token vocabulary, and `safeForSeedMutation=NO`.
3. Choose one canonical storage boundary:
   - a versioned metadata asset loaded into `RuntimeExerciseMetadataCatalog`, or
   - a separate Room metadata extension table keyed by `stableKey`.
4. Prefer the versioned asset first. It avoids a migration while taxonomy is still evolving and keeps unknown future tokens lossless.
5. Load the resource once in the repository/application scope and inject the catalog into `AnalysisInputCollector` and `ProgramSkeletonGenerator`.
6. Keep legacy fields read-only as fallback for old custom exercises and backups.
7. Add backup metadata version and stable-key references after the resource format is stable.
8. Run focused parser, seed/resource loader, program generator, backup restore, and analysis-input tests.

## Canonical Priority

Runtime consumers must prefer:

`canonical taxonomy -> compatibility adapter -> legacy fallback`

They must not infer canonical values from legacy fields when an approved canonical row exists.

## Deferred Analysis Work

For v0.3.6.0, connect preserved tokens to muscle-volume, stress/fatigue, recovery decay, badminton transfer, and chart interpretation modules. Unsupported tokens must remain safe no-ops until each calculator has focused tests.
