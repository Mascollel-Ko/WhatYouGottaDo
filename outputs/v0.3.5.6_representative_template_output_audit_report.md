## Rehab-Like Audit

No normal strength, transfer, or performance session contained more than one rehab-like activation item.

The initial audit flagged two rehab-like items in `STANDARD_5D_8W_70_30`, week 6, day 7:

- wall slide in `RECOVERY_PREHAB_LIGHT`
- band external rotation in `SCAPULAR_SHOULDER_SUPPORT`

That day is explicitly `RECOVERY_WEAKPOINT`, so the recovery-context allowance applies. The audit rule was corrected to distinguish explicit recovery sessions from normal performance sessions. No ProgramBuilder behavior change was required for this case.

All five programs remain below the existing 10-15% four-week rehab-like dominance warning range.

## Footwork / COD Audit

No representative program approached the 45% dominance cap. No tuning was required.

## Athletic Overhead Press Audit

### Before tuning

The resolver exposed 14 valid overhead candidates. Selection included useful half-kneeling presses but also rotated through generic machine, Arnold, pike, and handstand press variants. These were valid overhead movements, but the `ATHLETIC_OVERHEAD_PRESS_SUPPORT` slot did not sufficiently prefer trunk-integrated badminton-supportive options.

### Tuning

`ProgramCandidate.templateSpecificityFit()` now gives a small metadata-based bonus when an overhead candidate also has `TRUNK_ANTI_ROTATION_STABILITY` as a secondary capability. A smaller bonus remains for supportive-transfer candidates.

No exercise-name rule or blacklist was added.

### Final output

Selected overhead variants are now limited to the relevant half-kneeling family:

- one-arm half-kneeling press
- half-kneeling kettlebell one-arm press
- half-kneeling kettlebell press
- half-kneeling press

The exact subset varies by template and week. No rehab-like candidate filled the athletic overhead slot.

## Rotational Kinetic-Chain Audit

### Problem found

The original resolver produced three apparent rotation candidates:

- face pull
- 30-second repeated smash test
- wall slide

Face pull and wall slide were false positives caused by the broad `ROTATION_CONTROL` token matching scapular external/upward-rotation control metadata. The smash test contains rotation-power stress metadata but is `ANALYSIS_ONLY`, so it is not eligible for program generation.

The actual program-selectable rotation exercise, `ViPR downward twist`, was not recognized because its canonical family is the broad `POWER_CORE_SHOULDER` family and its subtype was not in the resolver rule.

### Tuning

- Removed broad `ROTATION_CONTROL` matching from `ROTATIONAL_KINETIC_CHAIN`.
- Added the approved existing subtype `VIPR_DOWNWARD_TWIST`.
- Did not change canonical metadata or create a new taxonomy value.
- Added a 14-day minimum repeat gap to rotational slots in the three templates that request them.

### Final output

The program-selectable rotational candidate pool now contains exactly one valid row:

- `바이퍼 트위스트(아래로)` / `ex_8354acd`

Exposure is therefore capped rather than falsely diversified:

| Template | Rotation weeks | Count |
|---|---|---:|
| `HYBRID_4D_4W_50_50` | 1, 3 | 2 |
| `STANDARD_5D_8W_70_30` | 1, 3, 5, 7 | 4 |
| `STRENGTH_BIASED_5D_8W_30_70` | 1, 3, 5, 7 | 4 |

Remaining limitation: the canonical catalog has only one valid program-selectable rotational kinetic-chain candidate. More variety requires future exercise/metadata additions and was not invented in this tuning pass.
